Key Take-Away Techniques (Cheat-Sheet)

  1. CTEs for static + dynamic metadata ‚Äì quick, readable lookup tables.

  2. Temp-table orchestration ‚Äì isolate every logical stage; simplify replay & debugging.

  3. Dynamic SQL ‚Äì generated once, cached in #QueryStore, executed in tightly-scoped cursor loops.

  4. Window functions (LEAD) ‚Äì ideal for before/after diffing without self-joins.

  5. GUID normalisation & reference-table discovery ‚Äì fully data-driven; no hard-coded FK lists.

  6. Collation-aware joins ‚Äì critical when CRM physical names differ in case or collation.

  7. StringMap resolution ‚Äì converts opaque integer option-sets to business-friendly text early, so later filters/readers see clear values.

  8. Defensive clean-ups ‚Äì apostrophe strip, IsAuditEnabled = 1, boolean coercion.

Performance / Maintainability Tips

  • Indexing ‚Äì consider a non-clustered index on Audit.CreatedOn + ObjectId when the audit table grows into millions.

  • Avoiding cursor row-by-row fetch ‚Äì possible rewrite with STRING_SPLIT (SQL 2016+) or cross-apply XML shredding if lower latency is required.

  • Temp-table sizes ‚Äì monitor in tempdb; large QI portfolios may drive spill.

  • Move ETL pieces into stored procedures ‚Äì keeps Confluence script concise while retaining a single entry-point for schedulers.

1 Why only the last 30 days?

  • At the very top we do

DECLARE @DateFrom DATETIME = GETDATE() - 30;
SET     @DateFrom = dbo.fn_LocalTimeToUTC_rpt(@DateFrom);

  • GETDATE() ‚Äì 30 ‚Üí rolling window: the report is always ‚Äútoday minus thirty‚Äù.

  • Converting to UTC up-front means we can safely compare with the CRM Audit.CreatedOn column, which is stored in UTC.
    If we stayed in local time we‚Äôd miss/twin some rows around daylight-saving edges.

  • ‚û°Ô∏è Grain of the report: one change to one field to one record inside that 30-day slice.

3 Exploding one audit row ‚Äì the cursor loop (Section 15)

Think of one Audit row as a zip-file that contains N pairs:
[ColumnNumber] & [OldValue].
The cursor unzips it like this:

  1. Pick the row from #temp (outer cursor variable @outeri).
    We only need its PK, the mask string and the old-value string.

  2. Split
    SQL Server 2008 doesn‚Äôt have STRING_SPLIT, so we cheat with

    INSERT #Mask(Item) VALUES ('...'),(...)  -- comma ‚Üí new row
INSERT #Vals(Item) VALUES ('...'),(...)  -- tilde  ‚Üí new row

    Building a string and executing EXEC(@Insert) is the pre-2016 way.

  3. Marry the two lists
    Join #Mask m ‚Üî #Vals v ON m.ID = v.ID ‚Üí you now have
    (ColumnNumber, OldValue) rows.

  4. Translate numbers ‚Üí field names
    Join to MetadataSchema.Attribute with ColumnNumber.
    ‚ûú we learn TargetFieldName, data type, whether it‚Äôs an Option-Set, a GUID, etc.

  5. Stash into #Atrib (one row per field that changed).

  6. Generate dynamic SELECTs (#QueryStore)

    SELECT <field> FROM <entity table>
WHERE  <PK> = '<objectId>'

    They will later fetch the current value (the ‚Äúnew‚Äù side).

  7. Copy everything into #AtribStore together with audit meta (user, time, action).

Why a cursor?
We could rewrite in set-based XML, but (a) CRM 4/2011 audit blobs are short, in-row strings, so a cursor is fast enough; (b) SQL 2008 has no native split function; (c) dynamic SQL must be executed row-by-row anyway.

6 Deduplicating to the single ‚ÄúOld ‚Üí New‚Äù row (Section 25)

LEAD(OldValue) OVER (PARTITION BY Name,FieldName ORDER BY CreatedOn)

  • The window function looks forward one row to pull the next OldValue,
    which is effectively the New value for the previous row.

  • If there is no later row (the most recent change) we fall back to the
    ‚Äúcurrent‚Äù value we fetched from the live table.

Result: one tidy row with OldValue and NewValue side-by-side, per change.

8 Key SQL Server 2008 techniques in play

  • Session-scoped temp-tables (#) + table variables (@Tab)
    for flexible scratch space.

  • Dynamic T-SQL with EXEC (@sql) when the column list itself is dynamic.

  • Cursors as a practical necessity for dynamic SQL per row.

  • Window functions (ROW_NUMBER, LEAD) ‚Äì introduced in SQL 2005, so fully available.

  • Extensive use of system catalog views (MetadataSchema, INFORMATION_SCHEMA) to stay data-driven ‚Äì no hard-coding of PKs or labels.

  • Collation awareness ‚Äì COLLATE DATABASE_DEFAULT every time we compare CRM logical names to audit text.

9 Why the design still works today

  • The CRM audit table hasn‚Äôt changed format since 4.0 ‚Äì string blobs are fixed.

  • The script relies only on T-SQL 2005‚Äì2008 features ‚Üí runs on anything later.

  • All ‚Äúwhat fields do we care about?‚Äù logic is in the CTE in Section 7 ‚Äì just add or remove ColumnNumbers there.

15.0 Why do we need a cursor at all?

  • Each record in #temp still looks like this:

ObjectId

AttributeMask

ChangeData

{‚Ä¶}

{10057,10059}

NZ~NZL

  • The mask and the value string are comma- and tilde-delimited snapshots of many columns that changed at the same instant.

  • We must split them, map the ColumnNumbers to real field names, and‚Äîlater‚Äîrun a dynamic SELECT for the new value of each field.

  • Because SQL Server 2008 doesn‚Äôt offer STRING_SPLIT() nor table-valued parameters for dynamic column names, a row-by-row cursor is the simplest safe choice.

15.1 Declarations & scratch tables

DECLARE @table         VARCHAR(MAX);   -- physical table name (e.g. AccountBase)
DECLARE @reporttable   VARCHAR(MAX);   -- alias ( = @table )
DECLARE @val           VARCHAR(MAX);   -- PK value of the record we‚Äôre dissecting
DECLARE @Atrib         VARCHAR(MAX);   -- raw mask string  ‚Äú10057,10059‚Äù
DECLARE @OldVal        VARCHAR(MAX);   -- raw old-value string ‚ÄúNZ~NZL‚Äù
DECLARE @PrimaryCol    VARCHAR(MAX);   -- name of PK column   (AccountId, ContactId ‚Ä¶)
DECLARE @OTC           BIGINT;         -- ObjectTypeCode

Why two table variables (#Mask, #Vals)?
We need to keep positional order: index 1 in #Mask must match index 1 in #Vals.

15.2 Outer cursor driver

SELECT @temprc = COUNT(*) FROM #temp;  -- how many audit rows?
WHILE @outeri <= @temprc

The loop processes one audit row per iteration.

Inside the loop we immediately:

  1. Reset scratch tables

    TRUNCATE TABLE #Atrib;
TRUNCATE TABLE #Mask;
TRUNCATE TABLE #Vals;

  2. Load variables for this row

    SELECT TOP 1
       @table       = TableName,
       @val         = ObjectId,
       @Atrib       = AttributeMask,
       @OldVal      = ChangeData,
       @OTC         = ObjectTypeCode,
       @reporttable = TableName,
       @PrimaryCol  = PrimaryColName
FROM   #temp
WHERE  ID = @outeri;     -- ‚Üê pointer of the outer loop

15.5 Prepare the dynamic SELECT for new values

For each row just loaded into #Atrib we:

WHILE @i <= @count
BEGIN
    SELECT @colm = TargetFieldName FROM #Atrib WHERE ID = @i;

    SET @s = 'SELECT ' + @colm +
             ' FROM ' + @reporttable +
             ' WHERE ' + @PrimaryCol + ' = ''' + @val + '''';

    INSERT #QueryStore
    SELECT @val,         -- ObjectId
           @colm,        -- column we will select
           @s,           -- full SQL text
           NULL;         -- placeholder for value later

    SET @i = @i + 1;
END;

Why store it instead of executing now?
Because we also create other rows later (e.g., from a second Audit row on the same object) and we want to batch-resolve all new values in Section 19 using one cursor.

15.6 Persist everything to #AtribStore

INSERT INTO #AtribStore
SELECT a.TableOTC,
       a.TableName,
       a.PrimaryColName,
       a.PrimaryColValue,
       a.TargetFieldNo,
       a.TargetFieldName,
       a.OldValue,
       a.CurrentValue,         -- NULL for now
       a.IsStringMap,
       a.IsGUID,
       a.DataType,
       a.ReferenceTableName,
       NULL, NULL,             -- PK & NameCol to be filled later
       a.ReferenceTableOTC,
       t.CreatedOnNZ,
       t.UserIdName,
       t.Action,
       NULL, NULL              -- dynamic GUID SQL placeholders
FROM   #temp   t
JOIN   #Atrib  a ON t.ObjectId = a.PrimaryColValue
WHERE  t.ID    = @outeri;

#AtribStore is now the master staging table for the rest of the script.

15.7 Advance the outer pointer

SET @outeri = @outeri + 1;

‚Ä¶and the loop begins again for the next Audit row until all have been processed.

Recap of Section 15 in one sentence

The cursor walks each Audit record, explodes its ‚Äúwhat changed?‚Äù blobs into tidy per-field rows, decorates them with metadata, builds the SQL needed to fetch their new values, and collects the whole lot in #AtribStore for later polishing.
