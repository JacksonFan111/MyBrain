Below is a detailed breakdown of the data relationships and hierarchy present in the query, along with an explanation of how the various tables and subqueries are joined to form the final dataset.

1. Base Entity – The Account Table

  • Table: CRM_MSCRM.dbo.Account (aliased as a)

  • Purpose:
    This is the primary or “base” table for the query. It holds the main trading entity information (accounts).

  • Key Field:

    • AccountId uniquely identifies each account.

  • Filter:

    • a.StateCode = 0 ensures that only active (current) accounts are included.
      (In Dynamics CRM, a StateCode of 0 conventionally indicates an active record.)

2. Portfolio Service Data

  • Table: CRM_MSCRM.dbo.dsl_portfolioservice (aliased as ps)

  • Join Condition:

    • Joined via ps.dsl_TradingEntityId = a.AccountId, meaning each account can have one or more portfolio service records.

    • Additionally, the filter ps.StateCode = 0 ensures only current (active) portfolio services are considered.

  • Key Fields Retrieved:

    • ps.dsl_PortfolioServiceId – a unique identifier for the portfolio service (combined with service name for granularity).

    • ps.dsl_PortfolioIdName and ps.dsl_PortfolioServiceLevelName – identifying portfolio details and level.

  • Additional Mapping:

    • A LEFT JOIN on CRM_MSCRM.dbo.StringMap (aliased as psu) on ps.dsl_PortfolioServiceStatus converts the numeric or coded status into a human‑readable text description.

3. Documentation and Tax Pool Data via StringMap

The query uses several joins to the StringMap table to convert coded values into user-friendly text:

  • W8 Documentation Status:

    • Join:

      • LEFT JOIN on CRM_MSCRM.dbo.StringMap (aliased as ds) using:

        • ds.AttributeValue = a.cip_W8DocumentationStatus

        • ds.AttributeName = 'cip_W8DocumentationStatus'

        • ds.ObjectTypeCode = 1

    • Purpose:

      • To display a descriptive status for W8 documentation.

  • W8 Current Tax Pool:

    • Join:

      • LEFT JOIN on CRM_MSCRM.dbo.StringMap (aliased as cp) using:

        • cp.AttributeValue = a.cip_w8currenttaxpoolnew

        • cp.AttributeName = 'cip_w8currenttaxpoolnew'

        • cp.ObjectTypeCode = 1

    • Purpose:

      • To convert the current tax pool code from the Account into a readable value.

  • Mapping of Account’s StateCode (Optional):

    • Join:

      • LEFT JOIN on CRM_MSCRM.dbo.StringMap (aliased as st) for converting a.StateCode into its textual equivalent.

    • Note:

      • Even though the main filter (a.StateCode = 0) already restricts the data to current records, this join can be used for reporting purposes (e.g., to display the actual status text).

4. Tax Pool Effective Transfer Details (Subquery Alias: tp)

  • Source Table:

    • ots_taxpooleffectivetransferdate

  • Purpose:

    • To obtain information on the tax pool transfer details including the previous tax pool, the new tax pool, and the transfer date.

  • Subquery Logic:

    • Joins to StringMap:

      • Two INNER JOINs to StringMap (aliased as sm and sm1) translate coded values for:

        • ots_Previousw8TaxPool

        • ots_NewW8TaxPool

    • Filtering:

      • The subquery includes WHERE tp.StateCode = 0 so that only current tax pool data is included.

    • Row Selection:

      • It uses ROW_NUMBER() partitioned by tp.ots_EntityId (which links back to AccountId) ordered by tp.ots_TransferDate DESC so that the most recent tax pool record (i.e., the latest transfer) for each account is selected (RN = 1).

  • Join to Main Query:

    • The subquery (aliased as tp) is joined on:

      • tp.ots_EntityId = a.AccountId

      • And filtered further with tp.RN = 1

  • Additional Use in Query:

    • The results from this subquery are used both for displaying tax pool values and for a CASE statement that checks for mismatches between the current tax pool (cp.value) and the new tax pool (tp.ots_Neww8TaxPool).

5. Advisor Information (Subquery Alias: advisor)

  • Purpose:

    • To fetch advisor-related details such as advisor code, advisor name, and commission branch.

  • Tables Involved:

    • dsl_advisercommissioncode (aliased as acc)

    • dsl_advisercommissionlink (aliased as acl)

    • SystemUser (aliased as su)

  • Subquery Logic:

    • Filtering:

      • The subquery ensures that:

        • The advisor is not retired (acc.dsl_Retired = 0).

        • The adviser commission link is active (acl.statecode = 0).

        • Pseudo advisers (e.g., those named 'HEADOFFICE') are excluded.

        • Advisor records are current (acc.StateCode = 0).

    • Join Conditions:

      • dsl_advisercommissioncode is joined to dsl_advisercommissionlink via acc.dsl_advisercommissioncodeId.

      • Then, it is joined to SystemUser via acl.dsl_Adviser to retrieve detailed user information.

  • Join to Main Query:

    • The advisor subquery (aliased as advisor) is joined on:

      • advisor.AdvisorID = a.dsl_AdviserCommisionCodeIdName (linking the advisor’s code from the account to the advisor record)

      • advisor.Individual = a.dsl_PrimaryAdviserId (ensuring the correct advisor is associated)

6. Overall Filtering and Data Integrity

  • Final WHERE Clause on Main Query:

    • a.StateCode = 0
      (Ensures only active/current accounts are included.)

    • ps.dsl_PortfolioServiceStatus <> '860910002'
      (Excludes portfolio services that are marked as “Closed”.)

    • ps.dsl_PortfolioServiceLevelName <> '303FB344-5F11-429C-921C-DEDA0387A754'
      (Excludes portfolio service records classified under a “Cash Management” code.)

  • Computed Column:

    • The CASE statement comparing cp.value (current tax pool) with tp.ots_Neww8TaxPool serves as an integrity check:

      • If the values differ (or if certain expected fields are missing), the result is flagged as 'MISMATCH'; if they are equal or meet the defined conditions, it is flagged as 'OK'.

7. Visual Hierarchy Summary (Bullet List)

  • Account (a)
    ├─ Portfolio Service (ps)
    │  └─ Mapped to Portfolio Service Status (psu) via StringMap
    ├─ Documentation & Tax Pool Attributes
    │  ├─ Mapped to W8 Documentation Status (ds) via StringMap
    │  └─ Mapped to W8 Current Tax Pool (cp) via StringMap
    ├─ Tax Pool Transfer Details (tp subquery)
    │  └─ Uses internal StringMap joins to map previous and new tax pool values
    └─ Advisor Information (advisor subquery)
    └─ Joins adviser commission code, link, and SystemUser to fetch advisor details

Conclusion

This query builds a comprehensive record by combining data from the primary Account table with associated portfolio services, tax pool transfer details, documentation statuses, and advisor information. Each join and subquery is designed to ensure that only the current, active records are considered (using StateCode = 0 and other filters), and that coded fields are translated into meaningful, readable values via the StringMap table. This relational hierarchy enables a detailed report that cross-references multiple facets of account data in a Dynamics CRM system.

Below is a table that maps out the custom columns added (in addition to the base columns from your SQL query) along with an explanation of the business logic used for each. This table is designed to help non‑technical users understand what each custom column represents and how its value is determined.

Custom Column Name

Explanation / Business Logic

QI30 Check

Checks if the current tax pool (TE_W8CurrentTaxPool) equals "QI30". If it does, the value is flagged as “INVALID - QI30” (indicating an invalid tax pool condition), otherwise it returns “OK”. This is a simple flag that quickly identifies records with a QI30 tax pool.

Documentation Status vs Tax Pool

Compares the current tax pool value and the documentation status to determine if they are consistent. The nested logic is as follows: • For tax pool QI15: It expects the documentation status to be “Documented and Transferred” – if not, the result is “MISMATCH”. • For tax pool QI30: If the documentation status is null (missing), then it is “OK”; otherwise it flags a “MISMATCH”. • For tax pool NQI: If the documentation status is “Documented and Transferred”, it is a “MISMATCH” (since that status isn’t expected for NQI), else “OK”. • If the tax pool is null: It checks if the documentation status is one of “No Documentation”, “Not Required” or null; otherwise, it flags “MISMATCH”. • For any other values, it returns “OTHER - TO INVESTIGATE”.

Expiry Date

Evaluates the W-8 documentation expiry date with several checks based on the tax pool and documentation status. For records with tax pool QI15:• If the “Documentation Status vs Tax Pool” is “MISMATCH”, it returns “OK” (perhaps as an exception condition). • If the documentation status is “Documentation Expired”, it returns “EXPIRED”. • If the year part of the expiry date equals 9999 (interpreted as a placeholder or no expiry), it returns “OK”. • If the expiry date is missing, it returns “EXP DATE MISSING”. • If the expiry date is within 120 days from the current date, it returns “DUE TO EXPIRE”. • If a validation date exists and the period between validation and expiry is 1461 days (or more), it flags “VALIDITY PERIOD TOO LONG”. For documentation statuses of “Not Required” or “No Documentation”, it checks if an expiry date is present and whether it’s later than today, returning either “CHECK DOC STATUS/TAX POOL” or “EXP DATE ISSUE”. Otherwise, it returns “OK”.

Validation Date

Checks the W-8 documentation validation date. For tax pool QI15 records, if the “Documentation Status vs Tax Pool” is “MISMATCH” then it returns “OK” as an override; otherwise, if the validation date is missing it flags “VAL DATE MISSING”, else “OK”. For records with documentation statuses “Not Required” or “No Documentation”, it similarly checks whether a validation date is present or if the expiry date is in the future (which would suggest that the document status and tax pool need a review), returning “CHECK DOC STATUS/TAX POOL” or “VAL DATE ISSUE” accordingly. In all other cases, it simply returns “OK”.

Exp Date Format Check

Ensures that when the tax pool is QI15, the W-8 documentation expiry date uses the expected day/month format. Specifically, it checks if the day equals 31 and the month equals 12 (i.e. the expiry date ends on December 31). If the expiry date is null, it returns “OK”. If the day or month does not meet the criteria, it flags “31/12 NOT USED - CHECK”. For other tax pool values, it simply returns “OK”.

Late Transfer

This flag identifies records where the documentation is in a “Documentation Validated - Pending Transfer” state and the tax pool is NQI. It then checks if the validation date is more than 14 days old (i.e. if the difference between today’s date and the validation date is greater than 14 days). If so, it returns “TRANSFER PENDING > 14 DAYS” indicating a delay; otherwise, it returns “OK”.

Summary of Base vs. Custom Columns

  • Base Columns from SQL: These are directly retrieved from the database and include identifiers (like PortfolioId, TE_Name, etc.), advisor information, documentation status, expiry and validation dates, tax pool values, and a computed column ([Current Vs New TaxPool]) that compares current tax pool values.

  • Custom Columns (Added via M Code): These are additional checks and validations applied in Power Query that build on the base data. They are designed to enforce business rules such as:

    • Validating tax pool consistency (QI30 Check, Documentation Status vs Tax Pool).

    • Ensuring documentation dates meet specific criteria (Expiry Date, Validation Date, Exp Date Format Check).

    • Detecting delays in processing transfers (Late Transfer).

This mapping should help users understand how the SQL output is further processed by the Power Query M code to yield a dataset that not only contains raw values but also business-rule validations and flags that support data quality and process monitoring.

Below are several DAX measures that you can add to your Power BI model (assuming your table is called QI_Clients and it has at least two columns: one for the account number (we’ll assume it’s named Account Number) and one for the status (we’ll assume it’s named Status)). These measures calculate the distinct count of account numbers for all rows and then for each specific status.

1. Total Distinct Count of Account Numbers

TotalDistinctAccounts =
DISTINCTCOUNT(QI_Clients[Account Number])

This measure gives you the total number of unique account numbers in the QI_Clients table.

2. Distinct Count for Each Status

a. No Documentation

NoDocumentationCount =
CALCULATE(
    DISTINCTCOUNT(QI_Clients[Account Number]),
    QI_Clients[Status] = "No Documentation"
)

b. Documented and Transferred

DocumentedAndTransferredCount =
CALCULATE(
    DISTINCTCOUNT(QI_Clients[Account Number]),
    QI_Clients[Status] = "Documented and Transferred"
)

c. Not Required

NotRequiredCount =
CALCULATE(
    DISTINCTCOUNT(QI_Clients[Account Number]),
    QI_Clients[Status] = "Not Required"
)

d. Documentation Expired

DocumentationExpiredCount =
CALCULATE(
    DISTINCTCOUNT(QI_Clients[Account Number]),
    QI_Clients[Status] = "Documentation Expired"
)

3. (Optional) Creating a Summary Table

If you prefer to see all of these counts in a single summarized table visual, you can create a calculated table using the following DAX:

StatusSummary =
SUMMARIZE(
    QI_Clients,
    QI_Clients[Status],
    "DistinctAccounts", DISTINCTCOUNT(QI_Clients[Account Number])
)

This StatusSummary table will have one row per status with the distinct count of account numbers for that status.

How to Use These in Power BI

  1. Adding Measures:

    • In Power BI Desktop, select your table QI_Clients in the Fields pane.

    • Click on New Measure and paste the DAX code for the measure you need.

  2. Creating a Visual:

    • Use a card visual to display individual measures like TotalDistinctAccounts.

    • Use a table or matrix visual to show the summary table (StatusSummary) if you want a side-by-side comparison of each status.

These measures will ensure you have both the total count and the distinct count of account numbers for each status based on the filter criteria you provided.

 

QI (QUALIFYING INTERMEDIARY) REGIME

  • Craigs ‘go live’ date as a QI with the IRD is 25 November 2024. This is a commercial decision (not an obligation), we are not responsible for existing on historical IRS claims.

  • This regime reduces the withholding tax rate for clients from 30% to 15%.

  • The 15% tax rate can only be offered from a go-forward position once a client is validated, and cannot be backdated.

  • Craigs uses the W-8 form route (IRS provided forms used to validate that client is eligible for the double tax treaty rate on US income).

  • Advisors CANNOT complete forms for clients. For non-individual clients advisors CANNOT select the form type (due to specific forms being required based on their IRD-recognised entity type & selecting the entity type may constitute tax advice).

  • Data in the forms MUST match CRM records.

  • Almost all W-8 forms expire 31st Dec, 3 years at the end of the current year of signing (i.e. signed 8/05/24, expiry is 31/12/27. If form expires then IRS expects Craigs to remove them from QI, until a new W-8 form is provided.

  • USE USA DATE FORMAT (MM-DD-YYYY)

  • QI is not suitable for every client (guidance is that US tax WH be at least USD500 p.a. ~USD1,600 gross dividend income pa)

  • QI intranet page: https://craigsip.sharepoint.com/Compliance/SitePages/Qualifying-Intermediary-(QI).aspx?csf=1&web=1&e=EBMU0Y
