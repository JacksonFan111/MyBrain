📖 Overview

This document serves as a reference for new team members joining the Data Team, explaining the New Issues application’s UAT and PROD mismatch, the Bitbucket repository structure, and the current deployment workflow.

History & Background

The New Issues system is a web application that interacts with an SQL Server 2008 database, providing an interface for Finance business users to track IPO New Stocks being published within the organization.

Over time, inconsistencies have arisen between the UAT database and the develop branch in Bitbucket, primarily due to manual deployments.

Cynthia, the New Issues application admin, discovered a data mismatch in UAT. She identified the issue because a report that reconciles three areas—for example:

➡ A + B = C

was no longer accurate, as she found that:

❌ A + B ≠ C

This discrepancy led her to raise questions about the integrity of the UAT data.

🕰️ Evolution of the Deployment Process

  1. Initial Setup (Pre-DevOps)

    • Originally, both UAT and PROD databases were manually managed with scripts.

    • Changes were made directly to the UAT database without a proper version control system.

  2. Introduction of Bitbucket & Azure DevOps

    • The team migrated source code to Bitbucket aroun 2021 Auhest not sure and started using branch-based workflows.

    • Azure DevOps was introduced for build automation, but not for database deployments. (According to Fabio)

  1. Current State (Manual UAT Database Updates)

    • Develop branch (Bitbucket) is up to date, but the UAT DB is outdated.

    • There is no automated deployment for UAT, meaning manual scripts have caused inconsistencies.

    • The Azure DevOps pipeline does not include UAT deployment, only builds.

📌 Key Issues

  • Naming Confusion: The Develop branch is actually the UAT branch, but people assume otherwise.

  • Outdated UAT Database ans its SPs:

    • The UAT database does not match the develop branch because updates are applied manually.

    • The Data Team members often update only specific portions of the code without checking out the full develop branch.

    • This results in partial updates, where only certain stored procedures or schema changes are applied, leaving the UAT database inconsistent.

  • No Tracking of UAT Deployments:

    • There is no clear record of when or how database updates were applied to UAT.

    • Currently, we do not have a version history of these updates, making it difficult to trace past changes.

    • The only way to track modifications is through SQL Server Management Studio (SSMS) logs, which show who made changes and when they were applied on the UAT server (New Issues DB).
