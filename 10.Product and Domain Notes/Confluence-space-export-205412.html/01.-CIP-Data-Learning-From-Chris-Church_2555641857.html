Our Client service model ERD hierarchy kind of looked like this

  1. Contacts Table

    • Description: This table stores information about the contacts.

    • Examples:

      • Contact 1: Chris

      • Contact 2: Jackson

  2. Role Table

    • Description: This table defines the roles associated with each contact. (One contacts might have multiple roles)

    • Examples:

      • Role 1: Client

      • Role 2: Online Client

  3. Accounts Table (Acc/Trading Entity (Te)) >. This is all the advisor cares about >. A 6 digits number

    • Description: This table holds information about various accounts and their associated roles.

    • Examples:

      • Account 1: Mr. Chris & Trust

      • Account 2: Mr. Jackson

  4. Portfolio Table ()

    • Description: This table contains details about the portfolio associated with each account.

    • Examples:

      • Portfolio for Mr. Chris

      • Portfolio for Mr. Jackson

Here's a more detailed breakdown of how these tables might be connected:

  • The Contacts table contains basic information about each contact.

  • The Role table assigns specific roles to these contacts (e.g., whether they are clients or online clients).

  • The Accounts table links the contacts with their respective roles and any additional account details.

  • The Portfolio table provides further details related to each account, such as investments or services.

Key Learnings

  1. We have some old DBs

  2. We are going to use a completely new ALOHA system, not sure would impact the integration much

  3. we are replacing the Chelmers DBs and some of the CRM functionality into ALOHA suite

  4. we going to have some issues modernising the client interface or even swapping to a better interface.

  5. We might want to move to an API based architecture so we could get data from real time

My Summary

  • Client Data: Names, addresses.

  • Transactional Data: Orders, contracts.

  • Market/Asset Data: Asset names, prices.

  • Data from on-prem SQLDBs (operational, CRM) ingested into a Data Lake, processed in Azure Synapse, modelled in on-prem SSAS (54GB), and output to BI interfaces (Power BI, Client Portal, SSRS Insights).

Future State:

  • Transition to SS&C ALOHA system.

  • Data flow: MongoDB (SS&C) ‚Üí ALOHA Data Store (ADS) ‚Üí R-ALOHA Data Store (RADS). ** R stand for (Replicated)

  • Potential integration challenges and need for API-based architecture for real-time data access.
