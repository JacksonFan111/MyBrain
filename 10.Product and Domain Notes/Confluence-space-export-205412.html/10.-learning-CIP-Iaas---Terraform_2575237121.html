Infrastructure as code workspace

High level Flow

Bitbucket → Pull Request → Synk → TerraForm Cloud platform → On prem TerraForm Agent → Deploy to Azure

 Workflow Overview

1. Code Development

  • Developers create feature branches from DEV and work on their tasks.

  • Commit frequently and rebase as necessary.

2. Pull Request and Review

  • Once a feature/task is completed, a PR is created.

  • The PR undergoes peer review and Synk security checks.

  • If approved, the PR is merged into the DEV branch.

3. Daily Ingest and Weekly Merges

  • Daily commits and updates are made into the DEV branch.

  • Every Friday: Changes are merged from DEV to STG.

  • Following Thursday: Changes are merged from STG to PRD.

4. Deployment

  • Terraform Cloud Platform and On-Prem TF Agents are used for deployments.

  • Deployments to Azure are triggered from the master branch PR merges.

  • Separate workspaces manage different environments, and any failed workspace is rerun until successful.

Understanding the Infrastructure Repositories, Pipelines & Process Graph

The graph represents the infrastructure setup and deployment process using Terraform, Git, and various cloud services like Azure, Cloudflare, and Auth0. Here’s a breakdown of what’s happening in the graph:

  1. Repositories and Branching:

    • CAF Enterprise Scale and Infrastructure Platform repositories have multiple branches (master, level0, prd, global, dr) that handle different environments and scopes of deployment.

    • Infrastructure Landing Zones and Direct Infrastructure repositories are branched for different environments and auto-approval stages.

    • Application-specific repositories (app-x-infrastructure, app-y-infrastructure) handle application-specific infrastructure code and changes.

  2. Approval Process:

    • Changes in the repositories require pull requests (PRs) which are subject to operations team approval before being merged into the master branch.

    • The approval process ensures that only reviewed and approved changes are applied to the infrastructure, maintaining security and stability.

  3. Merge and Trigger Plan:

    • Once the PRs are approved and merged into the master branch, they trigger a plan in the Terraform workflow.

    • This process is automated to ensure that any changes to the codebase are systematically deployed through a CI/CD pipeline.

  4. Terraform Apply (Deployment):

    • The approved and merged changes from various branches trigger the terraform apply command, which deploys the infrastructure changes to the cloud.

    • Deployments are made to Azure, Cloudflare, and Auth0 based on the configuration and code present in the repositories.

  5. User Principal CLI Authentication:

    • For the CAF Enterprise Scale and Platform repositories, user principal CLI authentication is used to authenticate and authorize the deployment process.

    • This step ensures that only authorized personnel can initiate deployments, adding a layer of security.

  6. Terraform Remote State:

    • The graph mentions the use of Terraform remote state, which allows multiple team members to work on the infrastructure without conflicts by storing the state file remotely.

    • This ensures consistency and accuracy of the deployed infrastructure.

Key Components and Their Functions

  1. Repositories:

    • CAF Enterprise Scale: Manages the enterprise-wide Azure setup and configurations.

    • Infrastructure Platform: Handles the core infrastructure required for applications.

    • Infrastructure Landing Zones: Specific zones within the infrastructure that are isolated and managed independently.

    • Direct Infrastructure: Direct changes and configurations for immediate infrastructure needs.

    • App-specific Repositories: Infrastructure requirements specific to individual applications.

  2. Branches and Approval:

    • Each repository contains multiple branches representing different environments and stages of deployment.

    • Pull requests are used to review and approve changes before merging them into the master branch.

  3. Deployment Workflow:

    • Trigger Plan: Automated plans are triggered when changes are merged.

    • Terraform Apply: Applies the changes to the specified cloud services, ensuring the infrastructure is updated as per the latest configuration.

  4. Security and Authentication:

    • User principal CLI authentication ensures secure access to deploy changes.

    • Operations team approval is required to maintain oversight and control over the infrastructure changes.
