Background:

Contacts: Kylie Frappell Custodial & Portfolio Reporting Manager

This article provides a clear process for addressing the data discrepancies observed between the ARM (Asset Register Maintenance) system, which uses the Fusion database, and the AXYS/Portfolio Reporting (AP) system. AXYS is an older in-house system, originally developed by Richard Newton and Hugh. Despite their departure, Richard still offers insights as needed, though the system's data flow can be convoluted.

In 2024, a Portfolio Reporting Administrator, Cherry Ninon, raised an issue regarding an incorrect sector code for CCP.ASX in AXYS. After investigation, it became clear that even though the sector information was correct in ARM (Fusion), AXYS incorrectly listed CCP.ASX as "International Equities" instead of "Australian Equities."

The root cause of the issue lies in the AXYS setup and its synchronization process, where changes made in the AXYS application are not fully reflected in the AACPR database (the AP/Portfolio Reporting side), which is used for daily reporting.

Key Terms:

  • ARM (Asset Register Maintenance): The system that interacts with the Fusion database and is considered the source of truth for securities information.

  • AXYS/Portfolio Reporting (AP): An in-house portfolio reporting system where discrepancies have been identified. It uses AACPR, AACAXYS, AACEI as DB and BizTalk to migrate data.

  • AACPR: The database where client data from AXYS and other systems (AACEI and AACPR) is stored and reported on.

  • BizTalk: The integration service that migrates and synchronizes data between the AXYS application and the AACPR database.

  • SEDOL (Stock Exchange Daily Official List): Unique identifiers for securities.

  • ISIN (International Securities Identification Number): Unique identifiers used to track securities.

Issue Summary:

The issue was identified with CCP.ASX (Asset ID: 134420), where its sector classification in AXYS was incorrectly listed as International Equities, while in Fusion (ARM), it was correctly listed as Australian Equities. This caused a discrepancy in portfolio reports. Even though AXYS data was updated later to CCP.LSE, the changes were not reflected in the AACPR database due to BizTalk's failure to synchronize the update.

Root Cause:

  1. Non-Unique Security Code Setup:

    • In AXYS, the combination of Security Code and Exchange Code is not unique. This means that the same Asset ID can be entered multiple times with different exchanges, leading to potential errors.

  2. User Entry Error:

    • The user initially entered CCP.ASX (Asset ID: 134420) with the incorrect sector International Equities in the AXYS system, and this data was migrated to AACPR through BizTalk, leading to duplicated entries.

  3. Manual Correction in AXYS:

    • After realizing the mistake, the user manually updated AXYS to change CCP.ASX to CCP.LSE with the correct sector International Equities. However, the changes did not propagate to the AACPR database due to an incomplete data sync via BizTalk.

  4. Mismatch in Reporting:

    • The ARM-AP Daily Report compares the sector data between Fusion (ARM) and AXYS (AACPR). Since the incorrect sector was still stored in AACPR, the report flagged a discrepancy between CCP.ASX and CCP.LSE.

Resolution Steps:

  1. Update AXYS Sector Classification:

    • The issue resides in the AXYS/Portfolio Reporting (AP) system. The correct sector classification should be set manually using the Daily Report Editor tool.

    • To fix the discrepancy, the following updates are needed:

      • Asset ID: 134420 should be changed from CCP.ASX (International Equities) to CCP.LSE with the correct sector Australian Equities in Daily Report Editor.

      • Additionally, update the incorrect SEDOL code 6287658 to 0433918 for CCP.LSE.

  2. Validation:

    • After making these updates in the Daily Report Editor, wait for the next data synchronization (usually overnight) for the changes to reflect in the AACPR database.

    • Validate the updated information in AXYS, and verify that the ARM-AP Daily Report shows the correct sector classification (matching Fusion data).

  3. Consult IT or Backend Team if the Sync Fails:

    • If the updates do not reflect after the overnight sync, consult the IT or backend team to check BizTalk configuration or other underlying issues in the data migration process.

Future Actions and Guidelines for Developers:

  • ARM (Fusion) as the Source of Truth: Always consider Fusion (ARM) as the authoritative source when investigating data discrepancies. Errors usually stem from the AXYS/Portfolio Reporting (AP) system.

  • BizTalk Sync: Be aware that BizTalk is responsible for syncing data from AXYS to AACPR. If discrepancies persist even after correcting the AXYS data, the issue may be with BizTalk's synchronization process.

  • Redirect User Tickets:

    • If similar discrepancies occur in the future, redirect the end user to their own team and provide them with this knowledge article for reference. They should use the Daily Report Editor to correct sector classification issues in AXYS.

    • Encourage users to consult their Confluence page for more guidance: Link to Confluence Space.

  • No Direct Database Access:

    • The data team does not have direct access to update the AACPR database. Any necessary updates must be made through the Daily Report Editor by the end user (e.g., Kylie Frappell from Custodial & Portfolio Reporting).

Example Case:

  • Asset: CCP.ASX (needs to be updated to CCP.LSE)

  • Asset ID: 134420

  • Incorrect Sector in AXYS: International Equities

  • Correct Sector in Fusion: Australian Equities

  • Required Action: Update Daily Report Editor to reflect the correct data and allow BizTalk to sync the changes.
